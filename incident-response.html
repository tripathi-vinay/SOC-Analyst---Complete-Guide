üéØ Incident Response Framework (NIST)

üö® Why Incident Response Matters

The average cost of a data breach is $4.45 million. Organizations with a well-tested incident response plan can reduce breach costs by up to $2.66 million and contain incidents 54 days faster.

NIST Incident Response Lifecycle:

Phase 1: Preparation
Establish IR capabilities and resources
  • Develop incident response policies and procedures
  • Train incident response team members
  • Deploy monitoring and detection tools
  • Create communication plans and contact lists
Phase 2: Detection & Analysis
Identify and analyze potential incidents
  • Monitor security events and alerts
  • Analyze suspicious activities
  • Determine incident scope and impact
  • Document initial findings
Phase 3: Containment, Eradication & Recovery
Stop the incident and restore operations
  • Contain the incident to prevent spread
  • Eradicate threats from the environment
  • Recover affected systems and services
  • Validate system integrity
Phase 4: Post-Incident Activity
Learn and improve from the incident
  • Conduct lessons learned meetings
  • Update policies and procedures
  • Implement preventive measures
  • Complete final incident documentation

Level 1 - Info

SLA: 4 hours

Low impact incidents handled by SOC Tier 1

Level 2 - Warning

SLA: 2 hours

Medium impact requiring Tier 2 analysis

Level 3 - Critical

SLA: 1 hour

High impact requiring senior analysts

Level 4 - Emergency

SLA: 15 mins

Critical incidents requiring immediate response

ü¶† Malware Incident Response

Immediate Actions (First 15 minutes)

  • Isolate infected systems from network
  • Preserve system state for forensic analysis
  • Identify malware type and behavior
  • Assess lateral movement potential
# Malware Incident Response Commands # 1. Network Isolation (Windows) netsh interface set interface "Ethernet" disabled # Disable network adapter to prevent spread # 2. Process Analysis Get-Process | Where-Object {$_.ProcessName -like "*suspicious*"} tasklist /svc | findstr "suspicious_process" # 3. Memory Dump for Analysis # Using WinPmem for memory acquisition winpmem.exe -o memory_dump.raw # 4. File System Analysis dir /s /a /od C:\ | findstr /i "malware suspicious" # Look for recently modified files # 5. Network Connection Analysis netstat -ano | findstr ESTABLISHED # Check for suspicious network connections # 6. Registry Analysis (Common malware locations) reg query HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Run reg query HKCU\SOFTWARE\Microsoft\Windows\CurrentVersion\Run # 7. Event Log Analysis wevtutil qe Security /f:text | findstr "4688 4689" # Check for process creation events # 8. Hash Analysis certutil -hashfile suspicious_file.exe SHA256 # Calculate file hash for threat intelligence lookup # 9. Safe System Shutdown shutdown /s /f /t 0 # Force shutdown to preserve evidence

üî¨ Forensic Analysis Steps

‚ö° DDoS Attack Response

Traffic Analysis and Mitigation

  • Identify attack vectors and source patterns
  • Implement rate limiting and traffic filtering
  • Coordinate with ISP and CDN providers
  • Monitor service availability and performance
# DDoS Attack Response Commands # 1. Traffic Analysis netstat -ntu | awk '{print $5}' | cut -d: -f1 | sort | uniq -c | sort -n # Count connections by IP address # 2. Identify Attack Patterns tcpdump -i eth0 -n | head -100 # Capture network traffic for analysis # 3. Block Attacking IPs (iptables) iptables -A INPUT -s 192.168.1.100 -j DROP # Block specific attacking IP # Block subnet range iptables -A INPUT -s 192.168.1.0/24 -j DROP # 4. Rate Limiting (iptables) iptables -A INPUT -p tcp --dport 80 -m limit --limit 25/minute --limit-burst 100 -j ACCEPT iptables -A INPUT -p tcp --dport 80 -j DROP # 5. Monitor Server Load top -b -n1 | head -5 iostat -x 1 5 # Monitor CPU and I/O usage # 6. Check Network Interface Statistics ifconfig eth0 # Look for dropped packets and errors # 7. Apache/Nginx Rate Limiting # Apache mod_limit_req LoadModule limit_req_module modules/mod_limit_req.so LimitRequestRate 10 burst=20 # 8. Cloudflare API (if using Cloudflare) curl -X POST "https://api.cloudflare.com/client/v4/zones/ZONE_ID/firewall/rules" \ -H "Authorization: Bearer YOUR_API_TOKEN" \ -H "Content-Type: application/json" \ --data '{"action":"challenge","filters":[{"expression":"ip.src in {192.168.1.0/24}"}]}'

üõ°Ô∏è DDoS Mitigation Strategies

  • Cloud-based Protection: Cloudflare, AWS Shield, Azure DDoS Protection
  • Rate Limiting: Implement per-IP connection limits
  • Geoblocking: Block traffic from suspicious countries
  • Anycast Networks: Distribute traffic across multiple locations
  • CDN Implementation: Cache static content to reduce origin load

üíæ Data Breach Response

Critical First Steps

  • Contain the breach and prevent further data loss
  • Preserve evidence for forensic investigation
  • Assess the scope and type of compromised data
  • Notify legal team and prepare for compliance requirements
# Data Breach Response Commands # 1. Identify Data Access Patterns # Database access logs analysis grep "SELECT.*FROM.*users\|customers\|employees" /var/log/mysql/mysql.log | tail -100 # 2. File Access Monitoring (Linux) # Check for recent file access find /sensitive/data/ -type f -atime -1 -ls # Files accessed in last 24 hours # 3. User Activity Analysis last -f /var/log/wtmp | head -20 # Recent user logins # 4. Network Traffic Analysis # Check for data exfiltration netstat -nap | grep :443 | wc -l # Count HTTPS connections (potential data upload) # 5. Database Integrity Check # MySQL mysqlcheck --all-databases --check-only-changed # PostgreSQL psql -c "SELECT * FROM pg_stat_user_tables WHERE n_tup_ins > 1000 OR n_tup_del > 1000;" # 6. File Integrity Monitoring # Compare against known good checksums find /sensitive/data/ -type f -exec sha256sum {} \; > current_checksums.txt diff baseline_checksums.txt current_checksums.txt # 7. Memory Dump Analysis (if system is compromised) # Create memory dump for analysis dd if=/proc/kcore of=/tmp/memory.dump bs=1M count=1000 # 8. Incident Documentation echo "$(date): Data breach detected - $(whoami) investigating" >> /var/log/security/incident.log # 9. Secure Evidence Collection # Create forensic image dd if=/dev/sda of=/external/forensic_image.dd bs=512 conv=noerror,sync

üìä Post-Incident Analysis and Reporting

üìà Key Metrics to Track

  • Mean Time to Detection (MTTD)
  • Mean Time to Containment (MTTC)
  • Mean Time to Recovery (MTTR)
  • Number of affected systems/users
  • Financial impact and recovery costs
  • Lessons learned and process improvements
# Post-Incident Analysis Script #!/bin/bash # Incident Analysis Report Generator INCIDENT_ID=$1 START_TIME=$2 END_TIME=$3 echo "=== INCIDENT ANALYSIS REPORT ===" echo "Incident ID: $INCIDENT_ID" echo "Analysis Date: $(date)" echo "Analyst: $(whoami)" echo "" # Calculate Response Times echo "=== RESPONSE METRICS ===" DETECTION_TIME=$(date -d "$START_TIME" +%s) CONTAINMENT_TIME=$(date -d "$END_TIME" +%s) TOTAL_DURATION=$((($CONTAINMENT_TIME - $DETECTION_TIME) / 60)) echo "Total Incident Duration: $TOTAL_DURATION minutes" # System Impact Assessment echo "" echo "=== AFFECTED SYSTEMS ===" # Query SIEM for affected hosts curl -X GET "https://siem.company.com/api/hosts?incident_id=$INCIDENT_ID" \ -H "Authorization: Bearer $SIEM_TOKEN" | jq '.affected_hosts[]' # Evidence Collection Summary echo "" echo "=== EVIDENCE COLLECTED ===" ls -la /evidence/$INCIDENT_ID/ # IOC Summary echo "" echo "=== INDICATORS OF COMPROMISE ===" echo "File Hashes:" cat /evidence/$INCIDENT_ID/file_hashes.txt echo "IP Addresses:" cat /evidence/$INCIDENT_ID/suspicious_ips.txt echo "Domain Names:" cat /evidence/$INCIDENT_ID/malicious_domains.txt # Recommendations echo "" echo "=== RECOMMENDATIONS ===" echo "1.
